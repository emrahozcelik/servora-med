# Servora-Med Slice 01 Secure Auth Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add secure email/password authentication, hashed opaque cookie sessions, role-aware identity, password change, production Origin/CORS protection, login rate limiting, first-admin bootstrap, development seed, and an accessible login shell.

**Architecture:** Auth domain behavior lives in `modules/auth/service.ts` behind an `AuthRepository`; PostgreSQL owns persistence and transaction boundaries. HTTP handlers translate cookies and payloads only. Node `scrypt` hashes passwords; SHA-256 hashes high-entropy session tokens before storage.

**Tech Stack:** Fastify 5, `@fastify/cookie` 11, `@fastify/cors` 11, `@fastify/rate-limit` 11, PostgreSQL, Node crypto, React 19, Vitest 4

## Global Constraints

- Raw passwords and session tokens are never stored or logged.
- Session cookie is `HttpOnly`, `SameSite=Lax`, `Path=/`, and `Secure` in production.
- Production unsafe requests require exact configured `Origin`.
- Login responses contain safe user data, never the raw token.
- Email login is globally case-insensitive unique.
- Password policy is 12 to 128 characters.
- Session TTL defaults to 8 hours and is configuration-controlled.
- Development seed refuses production execution.
- First-admin bootstrap refuses to run after any user exists.
- No JWT, refresh token, PIN, shift, OAuth, password-reset email, or multi-tenant selector.
- New production functions follow RED/GREEN TDD.
- Workspace is not a Git repository; do not initialize Git.
